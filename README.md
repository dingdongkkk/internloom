# InternLoom — Talent Matching Backend

RESTful backend for InternLoom's three-actor (student / company / admin) talent
matching platform. **FastAPI + PostgreSQL + SQLAlchemy + Alembic.**

## Run locally (~3 minutes)

Prereqs: Python 3.9+, Docker (for Postgres). No Docker? See "SQLite fallback" below.

```bash
# 1. Postgres
docker compose up -d

# 2. Install
cd backend
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt

# 3. Configure + migrate + seed
export DATABASE_URL=postgresql+psycopg2://internloom:internloom@localhost:5432/internloom
export JWT_SECRET=dev-secret-change-me
alembic upgrade head
python seed.py

# 4. Run
uvicorn app.main:app --reload
```

Interactive docs: http://localhost:8000/docs — health check: `GET /health`.

**SQLite fallback (no Docker):** `export DATABASE_URL=sqlite:///./internloom.db`
before step 3 — every feature works for browsing/demo, **except** the Bonus B
concurrency guarantee: SQLite has no row locks (`FOR UPDATE` is a no-op), so the
cap-race test must be run against Postgres. (Run against SQLite, the test
correctly *detects* the race — evidence it's a real test, not a rubber stamp.)

## Architecture

```
app/
  api/routers/      HTTP layer — Pydantic validation, envelope responses only
  api/middleware/   rate limit (Bonus A), audit trail (Bonus C)
  services/         business logic — matching, state machines, concurrency-safe apply
  models/           SQLAlchemy schema (normalized skills via join tables)
  core/             envelope, JWT, bcrypt, role guards, error handlers, config
  db/               engine/session + Base
alembic/            migrations (schema deliverable — autogenerated from models)
```

Routes never contain business logic; services never build HTTP responses. Every
response uses the envelope in `core/envelope.py`:
`{"success": bool, "data": ..., "error": {code, message, details}, "meta": {pagination}}`.

## Key design points (full answers in [docs/DESIGN_DECISIONS.md](docs/DESIGN_DECISIONS.md))

- **Matching** — query-time weighted score (required 0.45 / preferred 0.15 /
  branch-year 0.20 / completeness 0.12 / recency 0.08), returned with a per-signal
  breakdown so students see *why* they ranked. O(data-read), not O(n²).
- **Cap race** — `SELECT … FOR UPDATE` on the listing row + a DB unique constraint:
  5 concurrent applies to 1 slot → exactly one 201, four clean 409s.
- **Withdrawal reopen** — `closed_reason` distinguishes a company's manual close
  (irreversible) from a cap-driven auto-close (derived state, reopens on withdrawal).
- **Email change** — new address staged in `pending_email` + OTP; the old verified
  email keeps working, so re-verification never locks a student out.

## Submission artifacts

- Database schema: `backend/alembic/versions/` (migration = schema deliverable)
- Design decisions: [docs/DESIGN_DECISIONS.md](docs/DESIGN_DECISIONS.md)
- API collection: `postman/InternLoom.postman_collection.json` — 33 requests incl.
  a "judge tests" folder (personal-email 400, duplicate-apply 409, bad transition
  400, wrong-actor 403, malformed-body 422). Run the two Login requests first;
  they auto-store tokens.
- Concurrency proof (Bonus B): `backend/tests/test_cap_race.py` (run vs Postgres)
- End-to-end test suite: `backend/tests/test_smoke.py` — 38 checks over the whole
  API (auth, matching order, cap auto-close, withdrawal reopen, actor isolation,
  error envelopes, rate limit, audit). Runs in seconds with no server or DB setup:
  `PYTHONPATH=. python tests/test_smoke.py`
- Dashboard (Bonus D): `frontend/index.html` — a zero-build single-file SPA
  (open it in any browser with the API running; no npm). Student side: signup
  with OTP, profile editor with live completeness meter, matched roles ranked
  with per-signal score breakdowns, my-applications with withdraw. Company side:
  listing management with state transitions, post-a-role form, applicants ranked
  by frozen snapshot score with the legal status actions only, notifications
  with unread badge.
